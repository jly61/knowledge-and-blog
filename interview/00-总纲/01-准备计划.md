# 4. 项目一：银行内部流程审批管理系统

## 4.1 项目定位

这是主项目，用来证明 6 年 B 端复杂业务经验。

不要讲成普通管理系统，而要讲成：

> 银行内部多业务流程审批处理系统。

## 4.2 可以真实表达的内容

* 系统包含待办、已办、发起、知悉；
* 有统一流程处理页面；
* 左侧展示流程业务表单或业务列表；
* 右侧展示流程处理表单；
* 流程处理表单包含任务标题、下一处理步骤、下一处理人、处理意见；
* 不同流程的节点不一样；
* 下一处理步骤根据当前处理步骤判断；
* 下一处理人根据流程类型、处理步骤和特殊业务参数判断；
* 流程流转主要由后端实现；
* 前端负责流程处理交互、业务数据展示和处理参数提交；
* 批量审核时，后端生成虚拟业务 id；
* 流程处理页面通过虚拟业务 id 查询业务数据列表；
* 权限会受到菜单、URL、按钮、机构、角色、流程状态共同影响。

## 4.3 不要夸大的内容

不要说：

> 我设计了完整流程引擎。
> 我负责流程流转引擎。
> 我从 0 到 1 设计了审批平台。
> 我实现了低代码流程平台。

应该说：

> 我负责流程处理相关前端模块开发和维护。
> 我参与多流程页面复用和流程处理交互实现。
> 我负责根据流程类型、当前节点和业务参数处理下一步骤、下一处理人等前端逻辑。
> 流程流转由后端控制，前端负责处理交互和数据承载。

## 4.4 核心亮点一：多流程统一处理页面

### 普通说法

> 我做了流程处理页面。

### 推荐说法

> 我负责过多业务流程统一处理页面，不同流程在同一套处理框架下展示不同业务表单或业务列表，并根据流程类型、当前节点、业务参数动态控制下一步骤和候选处理人。

### 可写进简历

```text
负责多业务流程统一处理页面开发，页面左侧根据流程类型展示业务表单或业务列表，右侧统一承载任务标题、下一处理步骤、下一处理人和处理意见，支持不同流程节点下的处理交互复用。
```

### 面试可展开

* 不同流程如何区分？
* 下一处理步骤如何判断？
* 下一处理人如何加载？
* 单条审核和批量审核如何复用同一个处理页面？
* 流程流转由后端控制时，前端负责什么？

---

## 4.5 核心亮点二：下一步骤与下一处理人动态判断

### 推荐说法

> 不同流程的节点不一样，下一处理步骤不是固定写死的，而是需要前端结合当前步骤和流程类型判断。下一处理人也不是固定列表，而是根据流程类型、当前步骤、角色、机构或特殊业务参数获取。

### 可写进简历

```text
维护流程处理前端配置逻辑，根据流程类型、当前处理步骤和特殊业务参数控制下一处理步骤选项，并动态加载候选处理人，满足不同业务流程的差异化处理规则。
```

### 面试可展开

* 为什么下一步骤不能直接写死在页面里？
* 如果流程增加 subFlowType 怎么扩展？
* 下一处理人接口参数有哪些？
* 如果下一处理人为空怎么办？
* 查看态是否需要展示下一步骤和下一处理人？

## 4.6 核心亮点三：批量审核虚拟业务 id

### 普通说法

> 支持批量审核。

### 推荐说法

> 批量审核不是简单传多个 id，而是前端勾选多条数据后，由后端生成虚拟业务 id，流程处理页再通过虚拟业务 id 查询业务数据列表，让多条业务数据进入同一个流程处理上下文。

### 可写进简历

```text
支持批量审核场景，前端勾选多条业务数据后由后端生成虚拟业务 id，流程处理页基于虚拟业务 id 查询业务数据列表，实现多条业务数据在统一流程处理上下文中的展示与提交。
```

### 面试可展开

* 为什么不用直接传多个业务 id？
* 虚拟业务 id 解决了什么问题？
* 批量审核左侧为什么可能展示业务列表而不是单个表单？
* 批量审核和单条审核如何复用处理面板？
* 批量审核完成后列表如何刷新？

## 4.7 核心亮点四：动作权限治理意识

### 当前真实痛点

项目中权限不只是菜单权限和按钮权限，还会叠加：

```text
机构
角色
流程状态
数据状态
是否发起人
是否当前处理人
是否允许发起流程
```

因此很多页面容易出现大量复杂 `v-if`。

### 普通说法

> 我做了按钮权限。

### 推荐说法

> 项目里的权限不仅是按钮权限，还叠加了机构、角色、流程状态等业务规则。直接在模板中写大量 v-if 会导致逻辑分散、维护困难。更合理的方式是抽象成业务动作权限，例如 canCreate、canEdit、canDelete、canStartFlow，让页面只关心动作是否可用，具体规则收敛到策略层。

### 可写进简历的真实表达

```text
参与权限控制相关开发，基于菜单权限、URL 权限和按钮权限控制页面操作，并结合机构、角色、流程状态等业务规则判断新增、编辑、删除、发起流程等动作是否可用。
```

如果没有完成重构，不要写：

```text
完成动作权限策略层重构。
```

### 面试优化思路

```js
const actionRules = {
  edit: [hasUrlPermission, isSameOrg, isEditableStatus],
  delete: [hasUrlPermission, isDraftStatus],
  startFlow: [hasUrlPermission, isAllowedRole, isCanStartStatus],
}

function canAction(action, context) {
  const rules = actionRules[action] || []
  return rules.every(rule => rule(context))
}
```

## 4.8 核心亮点五：多开 tab 与精准刷新

### 当前真实场景

系统支持多开 tab，流程处理页和业务管理页可能同时打开。
如果简单使用全局 EventBus 发送 refresh，可能导致多个列表同时刷新，出现刷新范围不准确的问题。

### 普通说法

> 系统支持多开 tab。

### 推荐说法

> 多开 tab 的难点不只是打开多个页面，而是同一路由不同参数如何区分缓存实例、关闭 tab 后如何销毁缓存、详情页保存后如何精准刷新来源列表，以及流程页和业务页同时打开时如何避免全局刷新导致无关页面被动刷新。

### 可写进简历的真实表达

如果确实做过多开 tab 相关开发，可以写：

```text
参与多开 tab 场景下的页面状态维护，处理详情页、流程处理页与业务列表之间的数据刷新和缓存问题，保证不同业务页面在多实例打开时能够正确展示和更新数据。
```

如果只是遇到问题但没系统解决，可以面试讲优化思路。

### 面试优化思路

不推荐只发：

```js
eventBus.$emit('refresh')
```

更合理的是：

```js
eventBus.$emit('refresh', {
  businessType: 'FLOW_TASK',
  recordId: currentRecordId,
  sourcePage: 'flow-detail',
  targetPageKey: listPageKey,
  refreshScope: 'current-list'
})
```

或者：

```text
1. 通过 pageKey / tabKey 区分页面实例。
2. 事件携带业务类型、数据 id、来源页面、刷新范围。
3. 只刷新相关列表，不刷新无关页面。
4. 对 React / Next.js 项目，可以使用 query invalidation 控制服务端状态失效。
```

## 4.9 核心亮点六：审核体验优化

### 当前真实痛点

审核人员反馈：

```text
列表 -> 打开详情 -> 查看 -> 处理 -> 关闭 -> 回到列表 -> 再打开下一条
```

这个操作链路比较长，连续审核效率不高。

### 推荐说法

> 针对审核人员连续处理效率低的问题，我考虑过将列表 + 详情的操作方式优化为连续审核模式，例如详情页增加上一条 / 下一条，处理完成后自动进入下一条待办，或者采用左侧任务列表、中间业务详情、右侧流程处理的三栏布局，减少重复打开和关闭页面的成本。

### 注意

如果没有落地，不要写成项目成果。

可以在面试中这样表达：

> 这个优化方案当时没有完整落地，但这是我对审核场景体验优化的思考。

## 4.10 核心亮点七：复杂表单可维护性

### 当前真实场景

项目中存在字段较多的表单，部分表单项可达到几十个字段。

### 普通说法

> 我写过很多表单。

### 推荐说法

> 对于字段较多的业务表单，我会关注字段分组、默认值、校验、回显、编辑态 / 查看态切换、字典缓存、附件回显、提交参数转换等问题，避免所有逻辑堆在一个页面组件里。

### 面试优化思路

```text
页面层：
负责路由参数、整体状态、接口调用。

表单容器层：
负责数据加载、回显、提交、校验。

字段组件层：
负责具体控件渲染。

hooks / utils：
负责字典加载、权限判断、格式转换、默认值处理。
```

> 我主要负责银行内部 B 端流程审批相关模块。这个系统不是简单的增删改查，而是包含待办、已办、发起、知悉等任务视图，并通过统一流程处理页面兼容多个业务流程。页面左侧根据流程类型展示业务表单或业务列表，右侧统一处理任务标题、下一处理步骤、下一处理人和处理意见。

# 8. AI 应用深度补强：SSE 与 RAG

## 8.1 SSE 流式输出

### 它是什么

SSE，全称 Server-Sent Events，是一种服务端向浏览器持续推送文本数据的机制。浏览器通过 `EventSource` 或 `fetch` 读取流式响应，适合 AI 对话、AI 总结、AI 快读这类需要逐步返回内容的场景。

## 为什么需要

AI 接口生成内容通常耗时较长，如果等全部生成完再返回，用户会感觉页面卡住。
使用 SSE 可以让前端边接收边展示，提升体验。

## 和 WebSocket 的区别

```text
SSE：
- 服务端单向推送
- 基于 HTTP
- 更适合 AI 文本流式输出
- 实现简单
- 浏览器原生支持 EventSource

WebSocket：
- 双向通信
- 适合实时聊天、协同编辑、游戏、实时看板
- 协议和状态管理更复杂
```

AI 快读、AI 问答、AI 总结这类场景通常优先考虑 SSE。

## Next.js Route Handler 示例思路

```ts
export async function GET() {
  const encoder = new TextEncoder()

  const stream = new ReadableStream({
    async start(controller) {
      const chunks = ['正在分析内容...', '生成快读摘要...', '总结完成。']

      for (const chunk of chunks) {
        controller.enqueue(encoder.encode(`data: ${chunk}\n\n`))
        await new Promise(resolve => setTimeout(resolve, 500))
      }

      controller.enqueue(encoder.encode('data: [DONE]\n\n'))
      controller.close()
    },
  })

  return new Response(stream, {
    headers: {
      'Content-Type': 'text/event-stream',
      'Cache-Control': 'no-cache',
      Connection: 'keep-alive',
    },
  })
}
```

## 面试表达

> 对 AI 快读或 AI 问答这类生成时间较长的功能，我会考虑使用 SSE 做流式输出。相比一次性等待完整结果，SSE 可以让前端边接收边展示，提升用户体验。相比 WebSocket，SSE 更适合服务端单向推送文本内容，实现成本也更低。断线时可以结合重试、任务 id 和服务端缓存，让用户重新连接后继续获取结果或查看已生成内容。

## 8.2 RAG 基础流程

### 它是什么

RAG，全称 Retrieval-Augmented Generation，检索增强生成。
它不是模型训练，而是在大模型回答前，先从知识库中检索相关内容，再把检索结果和用户问题一起交给大模型生成回答。

## 为什么需要

大模型本身可能不知道企业内部知识、项目文档、业务制度、历史帖子。
RAG 可以让 AI 基于企业自己的知识库回答问题，减少胡编，提高答案可追溯性。

## 核心流程

```text
1. 文档收集
收集帖子、技术文档、制度文件、FAQ、项目文档。

2. 文档切分 Chunk
把长文档切成小段，方便检索。

3. 生成 Embedding
把每个 chunk 转成向量。

4. 存入向量库
例如 PostgreSQL + pgvector、Milvus、Pinecone 等。

5. 用户提问
用户输入问题。

6. 向量检索 Recall
把用户问题转成向量，从向量库中找相似 chunk。

7. 可选 Rerank
对召回结果重新排序，提升相关性。

8. 组装 Prompt
把用户问题和检索内容一起传给大模型。

9. 生成回答
大模型基于检索内容回答。

10. 引用来源 Citation
返回答案时附带引用来源，提升可信度。
```

## 关键概念

```text
Chunk：
文档切分后的文本片段。

Embedding：
把文本转换成向量表示。

Vector Search：
向量相似度检索。

Recall：
召回相关内容。

Rerank：
对召回内容重新排序。

Citation：
引用来源，让用户知道答案依据。
```

## 和技术交流社区的结合

技术交流社区中可以把以下内容作为知识库：

```text
帖子正文
技术文章
评论问答
官方文档
项目说明
常见问题
```

用户提问时：

```text
用户问题 -> 生成向量 -> 检索相关帖子 / 文档 -> 组装上下文 -> 大模型回答 -> 返回引用来源
```

## 面试表达

> 我理解 RAG 不是模型训练，而是通过检索外部知识增强大模型回答。比如在技术交流社区中，可以把帖子、技术文章、FAQ 切分成 chunk，生成 embedding 后存入向量库。用户提问时先做向量检索，找到相关内容，再把这些内容和用户问题一起交给大模型生成回答。这样可以让回答更贴近社区已有知识，也能通过引用来源降低 AI 胡编风险。

## 10.3 案例一：权限按钮异常

### 适用场景

某些用户看不到新增、编辑、删除、发起流程按钮，或不该看到的按钮出现。

### 可准备表达

```text
背景：
银行管理系统中，按钮展示不只依赖菜单 / URL 权限，还依赖机构、角色、流程状态等业务规则。

现象：
某个用户反馈按钮缺失，或者测试反馈不同角色下按钮展示不一致。

排查过程：
1. 先确认当前用户角色和机构。
2. 查看菜单权限是否分配。
3. 查看菜单绑定的 URL 权限是否存在。
4. 查看前端 v-permission 绑定的 code 是否正确。
5. 检查业务状态判断，例如流程状态、数据状态、是否当前处理人。
6. 对比同角色其他用户或同机构其他数据。

定位结果：
可能是 URL 权限未配置、按钮 code 绑定错误、机构角色判断条件遗漏，或者状态判断逻辑散落在页面 v-if 中导致规则不一致。

解决方案：
修正权限配置或前端判断逻辑，并补充对应角色、机构、状态下的回归用例。

复盘：
权限不应该只靠模板中的复杂 v-if，应该抽象 canEdit、canDelete、canStartFlow 等动作权限方法，统一维护规则。
```

## 10.4 案例二：流程提交失败

### 适用场景

审核提交失败、流程状态已变化、旧页面提交过期状态。

### 可准备表达

```text
背景：

现象：
用户提交审核时失败，提示状态不允许操作，或者后端返回流程状态已变化。

排查过程：
1. 查看请求是否正常发出。
2. 查看提交参数中的流程 id、业务 id、当前步骤是否正确。
3. 查看页面是否使用了缓存的旧数据。
4. 确认是否多个 tab 同时打开同一条数据。
5. 和后端确认当前流程真实状态。
6. 检查处理完成后页面是否刷新了最新流程数据。

可能是页面缓存了旧流程状态，用户在旧 tab 中继续提交，后端状态已经变化。

复盘：
多开 tab 场景下，流程状态是高风险数据，不能长期依赖旧缓存。关键操作前应校验状态，提交完成后应精准刷新相关页面。
```

## 10.5 案例三：多开 tab 刷新异常

### 适用场景

流程处理页和业务列表页同时打开，EventBus 全局 refresh 导致多个列表都刷新。

### 可准备表达

```text
背景：
系统支持多开 tab，详情页、流程处理页和业务管理列表可能同时打开。

现象：
某个页面保存或处理完成后，通过 EventBus 通知刷新，结果多个列表页面都刷新了，包括不应该刷新的页面。

排查过程：
1. 查看事件名是否过于通用，例如 refresh。
2. 查看事件参数是否携带业务类型、数据 id、来源页面。
3. 查看哪些页面监听了这个事件。
4. 确认页面是否能区分当前 tabKey / pageKey。
5. 检查处理完成后实际应该刷新的范围。

定位结果：
全局 EventBus 的事件粒度太粗，缺少业务上下文，导致多个页面响应同一个刷新事件。

解决方案：
刷新事件携带 businessType、recordId、sourcePage、targetPageKey、refreshScope 等上下文；监听方根据上下文判断是否需要刷新。

复盘：
全局事件不能只发 refresh，必须明确刷新范围。多开 tab 场景下需要通过 pageKey / tabKey 精准控制页面实例。
```

### 自检

* [ ] 我能讲清楚 EventBus 全局刷新的问题。
* [ ] 我能设计带上下文的刷新事件。
* [ ] 我能讲清楚 pageKey / tabKey 的作用。


### 面试表达

## 12.3 亮点包三：多页面刷新与缓存

### 目标

### 准备关键词

```text
tabKey
pageKey
keep-alive
sourcePage
businessType
recordId
refreshScope
精准刷新
```

### 面试表达

> 多开 tab 的难点在于同一路由不同参数如何区分页面实例，以及详情页或流程页操作完成后如何精准刷新来源列表。如果只通过全局 EventBus 发 refresh，可能导致多个无关页面刷新。更好的方式是事件携带业务类型、数据 id、来源页面、目标 pageKey 和刷新范围，或者用状态管理 / query invalidation 做精准失效。

## 22.3 短板回答

### 问：你主技术栈是 Vue2，会不会太旧？

> 我目前工作中主技术栈确实是 Vue2 和 Ant Design Vue，这是因为银行内部系统技术栈比较稳定。但我没有只停留在 Vue2，业余一直在补 Vue3、React、Next.js 和 TypeScript。
>
> Vue2 方面，我会重点把响应式、keep-alive、自定义指令、权限控制、多开 tab 这些项目相关问题讲深。Vue3 方面，我也在补 Composition API、Proxy 响应式、Pinia 和 Vite。
>
> 我希望下一份工作能去技术更开放的团队，把已有 B 端复杂业务经验和新技术结合起来。

### 问：你 React 到什么程度？

> React 我有公司项目和个人项目实践。公司项目中参与过 Pad 调查审核项目，主要是基于配置生成任务项和表单。个人项目中我用 React + Next.js 做过技术交流社区、博客、组件 demo 和移动端模板，完整走过页面、接口、数据库、AI 接入和部署流程。
>
> 我不会把自己包装成 React 专家，但 Hooks、组件状态、表单、列表、Next.js App Router、接口和部署这些我都在系统补强，并且能结合项目讲。

### 问：你不会 Java，怎么做主前端辅后端？

> Java 不是我的主语言，我不会把自己包装成 Java 全栈。我的后端能力主要是为了支撑前端和 AI 应用闭环，比如接口设计、数据库建模、鉴权、AI 服务调用、日志和部署。目前我更多用 Next.js、Node、Prisma、PostgreSQL 做实践。
>
> 对 Java 后端，我会补 Spring Boot 基础分层，至少能看懂 Controller、Service、DTO、VO 和接口流程，方便和后端协作。如果岗位要求深入 Java 后端开发，那不是我最匹配的方向；如果是主前端、需要一定后端理解和 AI 应用落地，我会更匹配。

### 问：你为什么想离开当前银行项目环境？

> 我现在的项目主要是银行内部管理系统，业务稳定，但技术环境相对封闭，技术栈更新比较慢。我希望下一阶段能去技术更开放的团队，在继续发挥 B 端复杂业务经验的同时，更多接触 Vue3、React、Next.js、TypeScript、AI 应用和工程化实践。
>
> 我不是因为现在项目没有价值才离开，而是希望自己的技术成长不要只停留在传统管理系统交付上，希望能进入更现代的前端工程环境。

### 问：你的沟通能力怎么样？

> 我不是特别外向型的沟通风格，纯口头讨论时有时容易遗漏细节，所以我会更依赖文档、流程图和会后记录来保证沟通闭环。
>
> 实际工作中，如果和业务或项目经理讨论了需求，我会把确认项、待确认项、风险点整理出来，避免口头沟通后遗漏。这个方式也帮助我在和后端、测试、同事协作时减少误解。
