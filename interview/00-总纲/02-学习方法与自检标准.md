# 面试准备学习方法与自检标准

## 1. 核心原则

准备面试不要只看资料，而要做到：

```text
能说出来
能写出来
能结合项目讲
能回答追问
```

我的学习方式应该是：

```text
项目驱动 + 问题驱动 + 输出驱动
```

不要平均学习所有知识点，先把项目讲清楚，再用技术点支撑项目，最后用案例证明能力。

---

# 2. 学习优先级

## 第一优先级：项目表达

先把这几个项目讲熟：

```text
1. 银行流程审批项目
2. React Pad 配置化项目
3. 企业级技术交流社区
```

尤其是银行流程审批项目，要能讲清：

```text
多流程统一处理
下一步骤动态判断
下一处理人动态加载
批量审核虚拟业务 id
权限复杂度
多开 tab / EventBus 刷新问题
审核体验优化思路
```

项目讲不清，技术点再多也用不上。

---

## 第二优先级：项目相关技术点

每个技术点都要和项目绑定。

示例：

```text
Vue2 响应式 -> 动态表单字段回显
keep-alive -> 多开 tab 缓存
v-permission -> 按钮权限
EventBus -> 列表精准刷新
SSE -> AI 快读流式输出
XSS -> 富文本和 AI 内容安全展示
RAG -> 技术社区知识问答
```

不要孤立背知识点。

---

## 第三优先级：通用前端基础

在项目主线清楚后，再补：

```text
JavaScript
TypeScript

浏览器
网络
安全
性能
工程化
测试与监控

```

# 3. 每个知识点怎么学

每个知识点都按 4 步学习。

## 第一步：解释清楚

回答：

```text
它是什么？
它解决什么问题？
项目里哪里会用到？
```

## 第二步：写最小 demo 或画流程

不要求复杂，但要能说明核心问题。

例如：

```text
v-permission demo
canAction 权限策略 demo
EventBus 精准刷新 demo
Next.js SSE demo
配置化查询表单 demo
```

## 第三步：绑定项目场景

每个技术点都要能说：

```text
在我的项目里，它对应什么问题？
```

## 第四步：整理成面试回答

最终输出一段 1 分钟左右的回答。

例如：

```text
我们系统支持多开 tab，如果只通过全局 EventBus 发 refresh，多个页面都可能被动刷新。我的优化思路是事件携带 businessType、recordId、sourcePage、targetPageKey、refreshScope，监听方根据上下文判断是否需要刷新，避免无关页面刷新。
```

# 4. 什么叫达到自检要求

每个知识点按 4 个等级判断。

## 等级 1：看过

```text
知道这个词，但解释不清。
```

不算通过。

## 等级 2：能解释

```text
能说清它是什么、解决什么问题。
```

基础通过。

## 等级 3：能写代码 / 画流程

```text
能写最小 demo，或者画出流程图。
```

技术通过。

## 等级 4：能结合项目回答追问

```text
能把它放到自己的银行流程项目、权限项目、AI 项目里讲清楚。
```

面试通过。

# 5. 每个知识点的通过标准

每个知识点都用这 4 个问题检查：

```text
1. 它是什么？
2. 为什么需要它？
3. 怎么实现？
4. 我的项目里哪里用过，或者可以怎么用？
```

四个都能答出来，才算通过。

示例：EventBus 精准刷新

```text
1. 它是什么？
EventBus 是一种事件通信方式，用于页面或组件之间通知变化。

2. 为什么需要？
详情页、流程处理页保存后，需要通知列表刷新。

3. 怎么实现？
事件中携带 businessType、recordId、sourcePage、targetPageKey、refreshScope。

4. 项目里怎么用？
流程处理页和业务列表页同时打开时，避免一个 refresh 事件导致多个无关列表刷新。
```

# 6. 什么叫达到产出要求

每周至少要有 4 类产出：

```text
1. 一份知识点笔记
2. 一个项目话术
3. 一个代码 demo 或设计方案
4. 一组自问自答
```

只看资料不算完成。

---

# 7. 合格产出标准

## 7.1 知识点笔记

合格结构：

```text
1. 它是什么
2. 为什么需要
3. 核心实现
4. 注意点
5. 项目关联
6. 面试回答
```

示例：

```text
Vue2 响应式：
- 核心是 Object.defineProperty
- getter 收集依赖，setter 触发更新
- 缺点是无法监听新增属性和部分数组变化
- 项目中动态表单字段回显要注意响应式问题
```

---

## 7.2 项目话术

每个项目至少准备 3 个版本：

```text
1 分钟版本：背景 + 职责 + 1～2 个亮点
3 分钟版本：背景 + 职责 + 难点 + 方案 + 优化思路
追问版本：为什么这么做、有什么问题、如果重做怎么优化
```

银行流程项目 3 分钟版本必须讲清：

```text
待办 / 已办 / 发起 / 知悉
统一流程处理页面
下一步骤和下一处理人
批量审核虚拟业务 id
权限复杂度
优化思路
```

---

## 7.3 demo 或设计方案

合格 demo 不要求复杂，但要包含：

```text
可运行代码或清晰流程
核心逻辑
简单说明
适用场景
不足
```

例如动作权限策略 demo：

```text
解决问题：
机构、角色、流程状态、URL 权限叠加导致 v-if 混乱。

核心思路：
页面只调用 canAction(action, context)，具体规则由策略函数维护。

示例动作：
edit / delete / startFlow
```

---

## 7.4 自问自答

不合格：

```text
Q：Vue2 响应式？
A：Object.defineProperty。
```

合格：

```text
Q：Vue2 响应式原理讲一下？

A：
Vue2 通过 Object.defineProperty 劫持 data 中的属性。getter 做依赖收集，setter 在数据变化时通知 watcher 更新视图。它的缺点是无法监听新增属性和部分数组变化，所以需要 Vue.set。

结合项目来说，如果动态表单回显时给对象新增字段，页面可能不会更新，所以要提前声明字段或使用 Vue.set。
```

# 8. 每周完成标准

每周结束时检查：

```text
知识点：
- 是否整理了至少 1 份知识点笔记？
- 是否能回答：是什么、为什么、怎么做、项目里怎么用？
- 是否至少 3 个知识点能闭卷讲出来？

项目：
- 是否整理了至少 1 个项目话术？
- 是否能 1 分钟讲出来？
- 是否能 3 分钟讲出来？
- 是否准备了至少 5 个追问？

代码 / 方案：
- 是否写了 1 个 demo 或方案？
- 是否能讲清它解决什么问题？
- 是否能讲清它的不足？

表达：
- 是否录音讲过一次？
- 是否听录音发现卡顿点？
- 是否修改了表达？
```

全部完成，才算本周达标。

# 9. 每天怎么学

## 工作日 1.5 小时

```text
20 分钟：看一个知识点
30 分钟：整理成面试回答
30 分钟：写 demo 或画流程
10 分钟：闭卷复述
```

每天只学 1 个点，不要贪多。

---

## 周末 3～5 小时

```text
1 小时：补知识体系
1 小时：写 demo 或设计方案
1 小时：项目话术
1 小时：自问自答
30 分钟：录音复盘
```

周末重点是整合和表达。

# 12. 最终达标标准

真正准备到位的状态：

```text
1. 面试官问项目，我能说 3 分钟不卡。
2. 面试官追问细节，我能答 3 层。
3. 面试官问原理，我能从定义、实现、项目场景讲。
4. 面试官问没做过的优化，我能明确说“项目中没有完整落地，但我的思路是”。
5. 面试官问短板，我不慌，能说明当前程度和补强方式。
6. 简历上的每一句，我都能解释清楚。
```

达到这 6 条，就算真正完成自检和产出要求。
