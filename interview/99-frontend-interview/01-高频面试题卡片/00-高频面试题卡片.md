# JavaScript

## 题目 1：new 做了什么？

### 一句话结论

`new` 的作用是创建一个新对象，把这个对象的原型指向构造函数的 `prototype`，执行构造函数并绑定 `this`，最后返回对象。

### 核心原理

`new` 主要做了四件事：

1. 创建一个空对象；
2. 让空对象的 `[[Prototype]]` 指向构造函数的 `prototype`；
3. 执行构造函数，并把 `this` 指向这个新对象；
4. 如果构造函数返回对象类型，则返回该对象；否则返回新创建的对象。

### 面试回答

`new` 本质上是一个对象创建和原型绑定过程。它会先创建一个空对象，然后把这个对象的原型指向构造函数的 `prototype`，接着用这个对象作为 `this` 执行构造函数。如果构造函数显式返回了一个对象，就使用这个返回对象；如果没有返回对象，就返回前面创建的新对象。

### 手写简化版

```js
function myNew(Constructor, ...args) {
  const obj = Object.create(Constructor.prototype)
  const result = Constructor.apply(obj, args)
  return result !== null && (typeof result === 'object' || typeof result === 'function')
    ? result
    : obj
}
```

### 项目关联

在日常 Vue / React 业务开发里不一定直接手写 `new`，但理解它有助于理解原型链、类组件、构造函数、实例方法和组件库内部对象创建方式。

### 常见追问

1. `Object.create` 做了什么？
2. 构造函数返回基本类型会怎样？
3. 构造函数返回对象会怎样？
4. 原型链和 `new` 有什么关系？

## 题目 2：闭包是什么？有什么应用和问题？

### 一句话结论

闭包是函数可以访问其定义时所在词法作用域中的变量，即使这个函数在外部被调用。

### 核心原理

JavaScript 使用词法作用域。函数创建时会记住它所在的作用域链。当内部函数被外部引用时，外层函数执行结束后，其变量不会被立即释放，因为内部函数仍然可能访问它们。

### 面试回答

闭包可以理解为“函数 + 它能访问的外部变量”。常见应用包括数据私有化、防抖节流、函数柯里化、模块封装等。它的问题是如果闭包长期持有大对象或 DOM 引用，可能导致内存无法释放，产生内存泄漏。

### 项目关联

在前端项目里，防抖、节流、缓存函数、事件回调、hooks 闭包问题都会涉及闭包。例如搜索框输入时使用防抖，本质上就是通过闭包保存上一次的定时器。

### 常见追问

1. 闭包为什么可能导致内存泄漏？
2. 防抖为什么用到了闭包？
3. React Hooks 里的闭包陷阱是什么？
4. 循环中使用闭包会出现什么问题？

# Vue

## 题目 3：Vue2 和 Vue3 响应式有什么区别？

### 一句话结论

Vue2 响应式基于 `Object.defineProperty`，Vue3 响应式基于 `Proxy`，Vue3 对新增属性、删除属性、数组、Map、Set 的支持更完整。

### 核心原理

Vue2 会在初始化时递归遍历 `data`，把已有属性转换成 getter/setter，所以对对象新增属性、删除属性、数组索引和数组 `length` 的监听存在限制。

Vue3 使用 `Proxy` 代理整个对象，可以拦截 `get`、`set`、`deleteProperty`、`has`、`ownKeys` 等操作，因此可以更自然地监听对象和集合类型的变化。

### 面试回答

Vue2 和 Vue3 最大的响应式差异在于底层实现。Vue2 使用 `Object.defineProperty`，只能对已有属性做劫持，所以新增属性需要 `Vue.set`，数组也需要重写 `push`、`pop`、`splice` 等方法。Vue3 使用 `Proxy` 代理整个对象，可以拦截更多对象操作，所以新增属性、删除属性、数组索引、Map、Set 等场景支持更好。

另外，Vue3 的 Composition API 更适合逻辑复用，可以把表单、权限、流程等逻辑抽成 composable，解决 Vue2 里 mixin 来源不清晰、命名冲突的问题。

### 项目关联

在 Vue2 的银行 B 端项目里，如果表单对象需要动态新增字段，通常要提前声明字段或使用 `Vue.set`。如果迁移到 Vue3，这类响应式限制会减少，但需要注意 `reactive` 对象解构后会丢失响应式，需要使用 `toRefs`。

### 常见追问

1. Vue2 为什么监听不到新增属性？
2. Vue2 数组为什么要重写方法？
3. Vue3 为什么用 Proxy？
4. Proxy 为什么要配合 Reflect？
5. reactive 对象解构为什么会丢响应式？

## 题目 4：Vue2 代码复用用什么？mixin 冲突优先级怎么处理？

### 一句话结论

Vue2 常用 `mixin` 做逻辑复用，但 mixin 容易产生命名冲突和来源不清晰的问题；冲突时通常组件自身选项优先，生命周期钩子会合并执行。

### 核心原理

Vue2 的 mixin 会和组件选项进行合并：

1. 生命周期钩子合并成数组，mixin 的钩子先执行，组件自身钩子后执行；
2. `data` 会进行合并，同名字段通常组件自身优先；
3. `methods`、`components`、`directives` 等对象选项冲突时，组件自身优先；
4. 多个 mixin 同时存在时，维护成本会升高。

### 面试回答

Vue2 中常见的代码复用方式有 mixin、extends、公共组件、renderless component 等，其中 mixin 使用最多。但 mixin 的问题是数据和方法来源不清晰，多个 mixin 叠加后容易出现命名冲突。冲突处理上，生命周期会合并执行，mixin 先执行，组件自身后执行；methods 等对象选项同名时，一般组件自身优先。

Vue3 里更推荐 Composition API，把复用逻辑抽成 composable，例如 `useTable`、`useForm`、`usePermission`，逻辑来源更清晰，也更适合 TypeScript。

### 项目关联

在银行 B 端项目中，查询条件、表格分页、按钮权限、流程处理等逻辑都可以复用。Vue2 阶段可以通过 mixin 或公共组件复用，但如果逻辑复杂，我更倾向于封装业务组件或工具函数，避免 mixin 堆叠导致维护困难。

### 常见追问

1. mixin 有什么缺点？
2. 生命周期冲突怎么处理？
3. data 冲突怎么处理？
4. Vue3 Composition API 为什么比 mixin 好？

## 题目 5：reactive 包装过的对象再包装一层 reactive 会发生什么？

### 一句话结论

Vue3 内部会缓存原对象和代理对象的关系，同一个对象重复调用 `reactive`，一般会返回已有的代理对象，不会重复创建多层代理。

### 核心原理

Vue3 响应式内部会通过 WeakMap 维护原始对象和代理对象之间的映射关系。当一个对象已经被代理过，再次调用 `reactive` 时，会优先返回已有代理，避免重复代理。

### 面试回答

如果一个普通对象被 `reactive` 包过，再次传入 `reactive`，Vue3 通常会返回同一个响应式代理对象，而不是再包一层。这样可以避免重复代理和性能浪费。Vue3 内部会通过缓存结构记录原对象和代理对象的关系。

不过需要注意，如果把响应式对象解构成普通变量，解构出来的值可能会丢失响应式追踪，所以复杂对象在 setup 中解构时通常要配合 `toRefs` 使用。

### 项目关联

在表单封装或状态管理中，如果多个函数都可能接收到同一个响应式对象，不需要担心被重复 reactive 包装成多层代理。但需要注意不要随意解构 reactive 表单对象，否则字段联动和校验可能出现响应式丢失问题。

### 常见追问

1. reactive 和 ref 的区别？
2. reactive 解构为什么会丢响应式？
3. toRefs 解决了什么问题？
4. readonly 再 reactive 会怎样？

# 工程化

## 题目 6：webpack 和 Vite 有什么区别？

### 一句话结论

webpack 是 bundle-based 构建工具，开发时通常需要先打包再运行；Vite 开发阶段基于原生 ESM 按需加载，启动和热更新更快，生产构建主要使用 Rollup。

### 核心原理

webpack 会从入口开始分析依赖，把模块打成 bundle。项目变大后，启动和重构建成本会升高。

Vite 开发环境利用浏览器原生 ESM，源码模块按需加载，依赖通过 esbuild 预构建，所以启动速度快。生产环境仍然需要打包，因为要做代码压缩、tree shaking、兼容性处理、资源合并和缓存优化。

### 面试回答

webpack 和 Vite 最大区别在开发构建机制。webpack 更偏传统打包模式，需要从入口构建依赖图并生成 bundle。Vite 开发阶段不需要把所有源码提前打包，而是利用浏览器原生 ESM 按需加载模块，同时用 esbuild 做依赖预构建，所以冷启动和热更新都更快。生产环境 Vite 仍然会用 Rollup 打包，目的是优化产物体积、代码分割和浏览器缓存。

### 项目关联

在企业后台中，如果项目模块多、页面多，webpack 启动慢和热更新慢会影响开发效率。Vite 更适合现代项目，但老项目如果是 Vue2 + Ant Design Vue 1.x，迁移时需要考虑插件兼容、语法兼容、环境变量和构建产物差异。

### 常见追问

1. Vite 为什么开发环境快？
2. Vite 生产环境为什么还要打包？
3. esbuild 和 Rollup 分别做什么？
4. webpack loader 和 plugin 区别？
5. Vite 怎么做代码分割？

## 题目 7：webpack 几种 hash 有什么区别？

### 一句话结论

`hash` 是整个构建级别的 hash，`chunkhash` 是 chunk 级别的 hash，`contenthash` 是文件内容级别的 hash，生产环境静态资源通常更推荐使用 `contenthash`。

### 核心原理

1. `hash`：一次构建生成一个统一 hash，任何文件变化都可能导致所有资源 hash 改变；
2. `chunkhash`：根据 chunk 内容生成 hash，某个 chunk 变化不会影响所有 chunk；
3. `contenthash`：根据最终文件内容生成 hash，适合 CSS、JS 等静态资源长期缓存。

### 面试回答

webpack 中 hash 主要用于浏览器缓存控制。`hash` 是整个构建维度，只要项目任意内容变化，所有文件名都可能变化，不利于长期缓存。`chunkhash` 是按 chunk 生成，适合 JS chunk。`contenthash` 是根据文件最终内容生成，内容不变文件名就不变，所以生产环境常用 `[contenthash]` 配合 CDN 和强缓存。

### 项目关联

如果企业后台发布新版本，希望用户加载到最新代码，同时未变化的静态资源继续走缓存，就需要合理使用 `contenthash`。如果使用不当，可能出现文件内容变化但文件名没变，导致用户仍然访问旧缓存的问题。

1. 为什么修改文件后文件名有时没变？

3. 浏览器强缓存和协商缓存区别？
4. index.html 应该强缓存吗？

# 网络与登录

## 题目 8：fetch 可以取消请求吗？和 axios 有什么区别？

### 一句话结论

fetch 可以通过 `AbortController` 取消请求；axios 是基于 Promise 的请求库，提供了拦截器、超时、自动 JSON 转换等更完整的工程能力。

### 核心原理

`fetch` 是浏览器原生 API，默认只在网络错误时 reject，HTTP 4xx/5xx 不会自动进入 catch，需要手动判断 `response.ok`。

`axios` 提供请求拦截器、响应拦截器、超时、请求取消、自动转换 JSON、统一错误处理等能力，更适合企业项目封装请求层。

### 面试回答

fetch 可以取消请求，方式是创建 `AbortController`，把 `controller.signal` 传给 fetch，需要取消时调用 `controller.abort()`。和 axios 相比，fetch 是浏览器原生能力，API 更底层；axios 在企业项目里更常用，因为它提供了拦截器、超时、统一错误处理、请求和响应转换等能力。比如 token 失效、401 拦截、请求重试这些，用 axios 封装会更方便。

### 示例

```js
const controller = new AbortController()

fetch('/api/list', {
  signal: controller.signal,
})

controller.abort()
```

### 项目关联

在表格查询、详情页切换、AI 生成停止等场景都需要取消请求。比如用户快速切换上一条/下一条数据时，可以取消上一次未完成请求，避免旧请求后返回覆盖新数据。

## 题目 9：token 被拦截怎么处理？

### 一句话结论

token 被拦截通常指接口返回 401/403，需要区分 token 过期、无权限、被踢下线等情况，再做刷新 token、重新登录或提示无权限处理。

### 核心原理

企业项目一般会在请求响应拦截器里统一处理登录态：

1. 401：未登录或 token 过期；
2. 403：已登录但无权限；
3. token 过期：尝试用 refresh token 刷新；
4. 刷新成功：重放原请求；
5. 刷新失败：清理登录态并跳转登录页；
6. 多个请求同时过期：需要请求队列，避免重复刷新 token。

### 面试回答

token 被拦截时，我会先看后端约定的状态码。如果是 401，一般表示登录态失效，可以判断是否有 refresh token。如果有，就发起刷新 token 请求，刷新成功后重放之前失败的请求；如果刷新失败，就清空本地登录信息并跳转登录页。如果是 403，则说明用户已登录但没有权限，应该提示无权限，而不是跳登录页。

需要注意并发场景：如果多个接口同时返回 401，不能每个接口都去刷新 token，需要加一个刷新锁，其他请求进入等待队列，刷新成功后统一重试。

### 项目关联

在银行后台项目中，菜单权限、按钮权限、接口权限都要和登录态配合。前端可以做体验层面的拦截和显隐，但真正的接口权限必须由后端校验，避免越权访问。

### 常见追问

2. 多个请求同时 401 怎么处理？
3. refresh token 放哪里？
4. token 放 localStorage 安全吗？
5. 前端权限是否可靠？

## 题目 10：什么情况下会发预检请求？为什么有时会发两次请求？

### 一句话结论

跨域请求中，如果请求不满足简单请求条件，浏览器会先发送一次 `OPTIONS` 预检请求，确认服务端允许后，再发送真正的业务请求。

### 核心原理

简单请求通常需要满足：

1. 方法是 `GET`、`POST`、`HEAD` 之一；
2. 请求头只包含 CORS 允许的简单请求头；
3. `Content-Type` 是 `application/x-www-form-urlencoded`、`multipart/form-data` 或 `text/plain`。

如果使用了 `PUT`、`DELETE`，或者携带 `Authorization`、自定义请求头，或者 `Content-Type` 是 `application/json`，通常会触发预检请求。

### 面试回答

预检请求是浏览器 CORS 机制的一部分。当跨域请求不是简单请求时，浏览器会先发一个 `OPTIONS` 请求，询问服务端是否允许当前源、方法和请求头。如果服务端返回允许，浏览器才会继续发送真正的请求。所以有时在 Network 面板里会看到两次请求，一次是 OPTIONS，一次是真实业务请求。

### 项目关联

企业后台接口通常会携带 `Authorization` token，并且常用 `application/json`，所以跨域场景下很容易触发预检。开发环境可以通过 Vite/webpack proxy 或 Nginx 代理避免浏览器跨域；生产环境通常需要后端正确配置 CORS。

2. 为什么携带 Authorization 会触发预检？
3. Cookie 跨域怎么处理？
4. `Access-Control-Allow-Origin: *` 能不能携带 Cookie？

# 安全与性能

## 题目 11：AI 生成的代码如何防御 XSS 攻击？

### 一句话结论

AI 生成内容不能直接信任，必须和用户输入一样按不可信内容处理，展示前做白名单过滤、转义或沙箱隔离。

### 核心原理

XSS 本质是恶意脚本被浏览器执行。AI 生成的 Markdown、HTML、代码片段如果直接通过 `v-html`、`innerHTML`、`dangerouslySetInnerHTML` 渲染，就可能执行恶意脚本。

### 面试回答

AI 生成的内容不能直接信任，要当成不可信输入处理。首先，Markdown 渲染时要限制原始 HTML，或者对 HTML 做白名单清洗；其次，富文本展示可以使用 DOMPurify 等库过滤危险标签和属性；第三，前端避免直接使用 `v-html` 或 `dangerouslySetInnerHTML` 渲染未处理内容；第四，可以配置 CSP 降低脚本执行风险；最后，后端入库前也要做校验和过滤，不能只依赖前端。

如果是展示 AI 生成代码，一般应该作为文本展示并做 HTML 转义，而不是把它当作 HTML 执行。

### 项目关联

企业技术交流社区里有发帖、Markdown、AI 快读、代码块展示，这些都涉及 XSS 风险。尤其 AI 快读是模型生成内容，也必须做安全处理。可以采用 Markdown 安全渲染、代码高亮前转义、后端敏感词和危险 HTML 校验、CSP 等组合方案。

1. v-html 有什么风险？
2. DOMPurify 做了什么？
3. CSP 能完全防住 XSS 吗？
4. token 放 localStorage 和 Cookie 哪个更安全？
5. 只做前端过滤够不够？

## 题目 12：如何提升 LCP？

### 一句话结论

提升 LCP 的核心是让首屏最大内容元素更早加载、更早渲染，通常从服务响应、资源加载、图片优化、JS 阻塞和渲染路径几个方面优化。

### 核心原理

LCP 衡量的是视口内最大内容元素完成渲染的时间。影响 LCP 的常见原因包括：

1. 服务端响应慢；
2. 首屏接口慢；
3. 首屏大图加载慢；
4. JS 包体积大，阻塞渲染；
5. CSS 阻塞渲染；
6. 字体加载慢；
7. 客户端渲染等待数据太久。

### 面试回答

优化 LCP 要先找出 LCP 元素是什么，可能是首屏大图、标题、卡片或 banner。然后针对性优化。如果是图片，就压缩图片、使用 WebP/AVIF、设置合适尺寸、加 preload 或 priority；如果是接口慢，就做接口合并、缓存、SSR/SSG 或骨架屏；如果是 JS 阻塞，就做代码分割、路由懒加载、减少首屏依赖；如果是服务端慢，就优化 TTFB 和 CDN。

### 项目关联

在企业技术交流社区首页，LCP 可能是首屏帖子列表或 Banner。可以通过减少首屏 JS、分页加载帖子、骨架屏、缓存热门帖子、图片懒加载和 SSR/预渲染来优化。企业后台项目则更关注首屏接口和组件库包体积。

4. 图片怎么优化？
5. SSR 为什么可能改善 LCP？

## 题目 13：了解微前端吗？不同项目间 CSS 样式冲突怎么办？

### 一句话结论

微前端是把一个大型前端系统拆成多个可独立开发、部署、运行的子应用；CSS 冲突可以通过命名空间、CSS Modules、Shadow DOM、样式前缀、构建隔离等方式解决。

### 核心原理

微前端常见方案包括 iframe、qiankun、single-spa、Module Federation、Web Components 等。它主要解决多团队协作、技术栈异构、独立部署和系统拆分问题。

CSS 冲突常见来源包括：

1. 全局选择器污染；
2. 组件库样式覆盖；
3. reset.css 冲突；
4. 多个子应用 class 名重复；
5. 样式加载顺序变化。

### 面试回答

微前端适合大型系统拆分，比如主系统集成多个子系统，每个子系统可以独立开发和部署。它需要处理路由隔离、样式隔离、JS 沙箱、应用通信和登录态共享等问题。

CSS 冲突可以分层处理：普通项目可以用 BEM、命名空间、CSS Modules、scoped；微前端场景可以给子应用统一添加样式前缀，或者使用 Shadow DOM 做更强隔离；如果是 iframe，则天然拥有独立 document，CSS 和 JS 隔离更彻底，但通信和体验成本更高。

### 项目关联

银行内部系统经常存在主系统和多个子系统集成。如果子系统技术栈不同，或者需要独立发布，可以考虑微前端。但如果只是少量页面嵌入，iframe 可能更简单。选型时要看团队规模、隔离要求、通信复杂度和部署方式。

3. Shadow DOM 能解决什么问题？
4. 主子应用如何共享登录态？
5. 子应用之间如何通信？

## 题目 14：AI 分析传输太快，除了节流控制还有什么方法？

### 一句话结论

除了节流，还可以通过缓冲队列、批量渲染、requestAnimationFrame 合并更新、后端分段 flush、虚拟列表、增量解析降级等方式降低 UI 渲染压力。

### 核心原理

AI 流式输出通常会不断返回 chunk。如果每收到一个 chunk 就立即 setState / 更新 DOM，可能导致频繁渲染、Markdown 重复解析、页面卡顿。

优化思路是把“网络接收频率”和“UI 渲染频率”解耦。

### 面试回答

AI 传输太快时，不能每个 chunk 都立即渲染。除了节流，可以在前端维护一个 buffer，把多个 chunk 先缓存起来，然后按固定时间片批量更新 UI。也可以用 `requestAnimationFrame` 合并到浏览器下一帧渲染。对于长文本，可以减少 Markdown 全量解析，改成阶段性解析或流结束后再完整解析。如果消息列表很长，还可以使用虚拟列表。后端也可以配合做分段聚合后再 flush，降低前端更新频率。

### 项目关联

企业技术交流社区的 AI 快读或 AI 分析，如果使用 SSE 流式返回，前端需要解析流数据并展示。如果更新太频繁，帖子详情页可能卡顿。可以用 buffer + 定时 flush 的方式，把网络流和 UI 渲染解耦，同时提供停止生成能力。

1. SSE 和 WebSocket 有什么区别？
2. chunk 边界为什么不等于 SSE 事件边界？
3. requestAnimationFrame 和 setTimeout 有什么区别？
4. Markdown 流式渲染有什么问题？
5. 如何实现停止生成？

## 题目 15：RAG 是什么？前端需要掌握到什么程度？

### 一句话结论

RAG 是检索增强生成，核心是先从知识库检索相关内容，再把检索结果作为上下文交给大模型生成答案；前端需要理解流程、交互、引用展示和权限边界。

### 核心原理

RAG 基本流程：

1. 文档解析；
2. 文本切分成 chunk；
3. chunk 生成 embedding；
4. 向量入库；
5. 用户提问；
6. 问题向量化；
7. 向量搜索召回相关 chunk；
8. 可选 rerank；
9. 拼接上下文；
10. 大模型生成答案；
11. 返回答案和 citation。

### 面试回答

RAG 可以理解为给大模型接一个企业知识库。模型本身不一定知道企业内部资料，所以系统会先根据用户问题去知识库里检索相关 chunk，再把这些内容放进 prompt 里，让模型基于检索结果回答。这样可以减少幻觉，并且通过 citation 告诉用户答案来源。

前端不一定要实现向量检索底层，但需要理解整体链路，能设计上传文档、解析状态、知识库管理、问答界面、引用来源展示、无答案提示、权限过滤和流式返回体验。

### 项目关联

企业技术交流社区可以把帖子、评论、技术文档切成 chunk，生成 embedding 后存入向量库。用户提问时，系统召回相关内容并生成回答，同时展示引用来源。对于企业系统，还要注意不同用户只能检索自己有权限看的内容。

### 常见追问

1. chunk 是什么？
2. embedding 是什么？
3. recall 和 rerank 区别？
4. citation 有什么意义？
5. RAG 能完全解决幻觉吗？
6. 企业知识库如何做权限控制？
